### 1. 핵심 질문

- 콜백 함수가 무엇인지, 그리고 콜백을 전달할 때 어떤 제어권이 함께 넘어가는지

### 2. 한 줄 요약

> 이 장의 핵심을 한 문장으로 말하면?

- 콜백 함수는 다른 함수의 인자로 전달되는 함수로, 호출 시점/인자/this에 대한 제어권도 같이 위임하게 된다.

### 3. 개념 구조 정리

#### 📚 핵심 개념

- 콜백 함수
- 제어권 위임
- 콜백 지옥

#### 🔗 개념 연결 구조

- 콜백 함수 : 다른 함수의 인자로 전달되는 함수
- 제어권 위임 : 콜백 함수를 전달할 때 아래 제어권도 같이 전달됨
  - 호출 시점 : 언제 실행할지 결정
  - 인자 : 어떤 값을 넘길지 결정
  - this : 어떤 객체를 바인딩할지 결정
    - 메소드를 콜백으로 넘기면 일반 함수로 동작함
    - 콜백을 실행하는 함수가 this를 명시적으로 바인딩할 수도 있음
- 콜백 지옥 : 콜백을 중첩해서 작성하여 들여쓰기가 과도해지는 현상
  - 최근에는 Promise + async/await로 해결함
    - Promise로 개선 (제어 흐름을 구조화)
    - async/await로 가독성 개선 (문법적 단순화)

### 4. 나의 언어로 설명해보기

콜백 함수는 다른 함수에게 언제, 어떤 값으로, 어떤 this로 실행할지 넘긴다.

### 5. 헷갈렸던 부분 / 오해했던 부분

#### ❓ 처음에 이해 안 됐던 점

- 객체의 메소드를 넘기면 this가 유지될 것이라 생각했다.

#### 💡 지금은 이렇게 이해했다

- 메소드를 콜백으로 넘기면 일반 함수로 호출되기 때문에 this가 깨질 수 있다.

### 6. 코드로 확인하기

#### 제어권 위임 확인

```js
function repeat(callback) {
  for (let i = 0; i < 3; i++) {
    callback(i);
  }
}

repeat((i) => console.log(i));
```

- 결과 : 0, 1, 2
- 이유 :
  repeat 함수가 콜백의 실행 시점과 인자를 모두 결정  
  → 제어권이 repeat에게 있음

#### 메소드 참조 전달

```js
const obj = {
  x: 10,
  getX() {
    return this.x;
  },
};

setTimeout(obj.getX, 0);
setTimeout(() => obj.getX(), 0);
```

- 결과 : undefined, 10
- 이유 :  
  첫 번째 setTimeout은 함수 참조를 전달하여 일반 함수로 호출되므로 undefined가 됨  
  두 번째 setTimeout은 콜백 내부에서 `obj.getX()` 형태로 메소드 호출이 이루어지기 때문에 this가 obj로 유지됨

#### forEach thisArg

```js
const obj = { x: 10 };

[1].forEach(function () {
  console.log(this.x);
}, obj);
```

- 결과 : 10
- 이유 : forEach 함수의 thisArg로 obj를 넘겨주어 명시적으로 this를 obj로 바인딩하게 됨

#### 콜백 vs 화살표 함수

```js
const obj = {
  x: 10,
  f() {
    [1].forEach(function () {
      console.log(this.x);
    });
    [1].forEach(() => {
      console.log(this.x);
    });
  },
};

obj.f();
```

- 결과 : undefined, 10
- 이유 :
  첫 번째 콜백은 일반 함수로 전달되어 forEach 내부에서 일반 함수로 호출되므로 default binding이 발생함  
  두 번째 콜백은 화살표 함수로 정의되어 f의 this(obj)를 그대로 사용함

#### setTimeout과 this

```js
const obj = {
  x: 10,
  getX() {
    console.log(this.x);
  },
};

setTimeout(obj.getX, 0);
```

- 결과 : undefined
- 이유 : 콜백으로 전달되며 일반 함수로 실행됨

### 7. 실무 연결 포인트 💼

#### 🔎 이 개념이 실제로 왜 중요한가?

- 이벤트 핸들러에서 this가 깨지는 문제
- setTimeout / Promise 콜백에서 this 유실

#### 🛠 앞으로 코드 작성 시 적용할 점

- 메소드를 콜백으로 넘길 때는 this가 깨질 수 있음을 항상 인지
- 필요하면 bind 또는 화살표 함수 사용
- 비동기 흐름은 Promise / async-await로 관리

### 8. 면접 대비용 정리

#### ❓ 예상 질문

1. 콜백 함수란 무엇인가요?

- 콜백 함수란 다른 함수의 인자로 넘겨지는 함수를 말합니다.  
  콜백 함수는 함수의 호출 시점, 함수의 호출 인자, this의 제어권을 전달 받은 함수에게 넘기게 됩니다.

2. 콜백 함수에서 this는 어떻게 결정되나요?

- this는 콜백이 어떻게 호출되느냐에 따라 결정됩니다.  
  대부분의 경우 콜백은 일반 함수로 호출되기 때문에 this가 바인딩되지 않고,  
  필요한 경우 bind나 thisArg로 명시적으로 지정할 수 있습니다.  
  화살표 함수는 상위 스코프의 this를 그대로 사용합니다.

3. 콜백 지옥이란 무엇이고 어떻게 해결하나요?

- 콜백 지옥은 콜백 함수가 중첩되면서 코드의 가독성과 유지보수성이 크게 떨어지는 문제입니다.  
  이 문제의 본질은 비동기 작업의 실행 흐름에 대한 제어권이 콜백을 호출하는 쪽에 있기 때문에  
  코드의 흐름을 예측하고 제어하기 어려워진다는 점입니다.  
  이를 해결하기 위해 Promise가 도입되었습니다.  
  Promise는 비동기 작업의 결과를 객체로 표현하여,  
  then 체이닝을 통해 흐름을 순차적으로 구성할 수 있게 합니다.  
  하지만 then 체이닝이 길어지는 경우 가독성이 다시 떨어질 수 있기 때문에,  
  async/await를 사용하여 비동기 코드를 동기 코드처럼 표현할 수 있게 되었습니다.

## 최종 인사이트

콜백 함수는 단순히 함수를 전달하는 개념이 아니라,  
함수의 실행 시점, 인자, this에 대한 제어권을 넘기는 것이다.

이로 인해 코드의 실행 흐름을 직접 제어하기 어려워지는 문제가 발생하며,  
특히 비동기 상황에서는 예측이 어려워진다.

Promise는 이러한 제어권 문제를 해결하기 위해 등장했으며,  
async/await는 이를 더 직관적으로 표현하기 위한 문법이다.
