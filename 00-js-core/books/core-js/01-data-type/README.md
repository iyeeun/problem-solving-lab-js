## Deep Dive

### 1. 핵심 질문

> 이 장은 결국 무엇을 설명하려는가?

- JavaScript에서 데이터가 메모리에 어떻게 저장되고 관리되는가?
- Primitive type과 Reference type이 왜 다르게 동작하는가?
- 불변성이란 무엇이며 왜 중요한가?

### 2. 한 줄 요약

> 이 장의 핵심을 한 문장으로 말하면?

JavaScript에서 변수를 선언하면 메모리의 빈 공간에 식별자를 저장하고, 값은 별도의 데이터 영역에 저장된다.  
변수에는 해당 데이터 영역의 참조가 저장된다.  
Primitive type은 값 자체를 저장하고 있으며,  
Reference type에는 객체가 저장되어 있고 객체 내부 프로퍼티가 다시 값을 참조한다.

### 3. 개념 구조 정리

#### 📚 핵심 개념

- Primitive type
- Reference type
- Immutable
- Reference value
- Garbage Collection
- Shallow copy / Deep copy
- Call by Sharing

#### 🔗 개념 연결 구조

데이터 타입  
-> Primitive type & Reference type  
-> 값 저장 방식 차이  
-> 불변성 유지 전략

- 메모리 효율성을 위해 변수와 데이터를 분리 저장
- Primitive는 값 자체가 불변하므로 안전하게 복사 가능
- 객체는 참조를 공유하므로 의도치 않은 변경 가능
- 따라서 불변 객체 패턴으로 예측 가능한 코드 작성

### 4. 나의 언어로 설명해보기

JS는 변수와 데이터를 나눠서 저장하여 메모리 효율을 높인다.  
Primitive type은 데이터 영역에 값을 저장한다.  
이 값은 immutable하여 변하지 않고, 새로운 데이터 영역을 참조하는 방식으로 변경된다.  
Reference type은 데이터 영역에 객체 참조를 저장하고, 내부 프로퍼티가 또다시 데이터 영역을 참조하는 형태이다.  
따라서 객체를 복사하면 동일한 객체를 참조하게 되기 때문에 내부 프로퍼티가 변경되면 두 객체에 동일하게 적용된다.

### 5. 헷갈렸던 부분 / 오해했던 부분

#### ❓ 처음에 이해 안 됐던 점

- Primitive 변수는 값을 직접 저장한다고 생각했다.
- 배열의 `empty`와 `undefined`의 차이를 처음에는 이해하지 못했다.

#### 💡 지금은 이렇게 이해했다

- 변수는 값의 참조를 저장하고 있으며, 해당 값에서 타입에 따라 값 혹은 참조를 저장하고 있다.
- 배열의 `empty`는 값이 없는 것이 아니라 해당 index 프로퍼티 자체가 존재하지 않는 상태이다.

### 6. 코드로 확인하기

```js
const obj1 = { a: 1 };
const obj2 = obj1;

obj2.a = 10;

console.log(obj1.a); // 10
```

#### 🔍 실행 결과

- `10`

#### 📎 왜 이런 결과가 나오는가?

- `const obj2 = obj1;`을 하면서 obj2에 obj1를 참조하게 되었고, 해당 `a` 프로퍼티를 `10`으로 변경하였기 때문에 동일한 객체를 참조하고 있는 `obj1`의 값도 변경되었다.

### 7. 실무 연결 포인트 💼

#### 💥 실무에서 이런 상황을 만난 적이 있다

```js
let str = 'hello world';
console.log(str); // hello world
str[0] = 'H';
console.log(str); // hello world
```

#### 🔎 이 개념이 왜 중요했는가?

- 문자열은 immutable하기 때문에 조작해도 반영되지 않았다.

### 8. 면접 대비용 정리

#### ❓ 예상 질문

1. Primitive type과 Reference type의 차이는?  
   Primitive type은 값 자체가 메모리에 저장되고 불변하며, 복사 시 값이 독립적으로 복사됩니다. 반면 Reference type은 객체가 저장된 메모리 주소를 참조하며, 복사 시 참조만 복사되어 같은 객체를 가리킵니다. 따라서 primitive는 재할당 시 새 값이 생성되지만, 객체는 내부 프로퍼티를 직접 변경할 수 있습니다.

2. React에서 불변성을 지켜야 하는 이유는?  
   React는 Virtual DOM 비교 시 얕은 비교(참조 비교)를 사용하여 성능을 최적화합니다. 객체를 직접 수정하면 참조가 동일해 React가 변경을 감지하지 못해 리렌더링이 일어나지 않습니다. 따라서 spread 연산자나 map, filter 같은 불변 메서드로 새 객체/배열을 생성해야 합니다. 또한 불변성은 이전 상태 추적, 순수 함수 작성 등을 가능하게 합니다.

3. JavaScript의 함수 인자 전달 방식은?  
   JavaScript는 Call by Sharing 방식입니다. Primitive는 값이 복사되어 전달되고, 객체는 참조의 복사본이 전달됩니다. 따라서 함수 내부에서 객체의 프로퍼티는 변경할 수 있지만, 새로운 객체를 할당해도 외부 변수에는 영향을 주지 않습니다.

## 최종 인사이트

- JavaScript에서 데이터를 저장하는 방식을 이해해야 버그를 예방할 수 있으며, 성능 또한 최적화할 수 있다.
- 불변성을 통해 예상 가능하고 안정적인 코드를 작성할 수 있다.
- Structural Sharing으로 성능과 불변성을 동시에 잡을 수 있다.
