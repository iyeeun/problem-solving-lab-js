# [완주하지 못한 선수](https://school.programmers.co.kr/learn/courses/30/lessons/42576)

## 문제 요약

- 참가자 목록과 완주자 목록이 주어짐
- **한 명만 완주 못함**
- **동명이인 존재 가능**
- 완주 못한 선수 이름 반환

## 입력 구조

- participant: 최대 100,000
- completion: participant.length - 1
- 이름 길이: 최대 20
- 중복 이름 가능

## 문제 유형 분류

- Hash (해시맵)

## 사고 과정

### 1. 왜 이 유형인가

- “누가 빠졌는가” → **카운팅 문제**
- 동명이인 존재 → 단순 `includes` 불가
  → **이름별 등장 횟수 관리 필요**

=> 해시가 가장 적합

### 2. 왜 다른 접근은 안되는가

- `includes`, `indexOf` → 시간 초과 (`O(n^2)`)

- Set → 중복 처리 불가

- 정렬 → O(n log n) 가능하지만 최적은 아님

## 문제 풀이

1. [42576.js (2분 소요)](./42576.js)

```
1. participant를 순회하며 카운트 증가
2. completion을 순회하며 카운트 감소
3. 값이 1 남은 사람이 정답
```

## 해결 전략

### 알고리즘 선택 이유

- 카운팅 기반 문제 : 한 번씩만 순회하면 해결 가능

### 자료구조 선택 이유

- Object 사용
  - key: 이름
  - value: 등장 횟수
  - 빠른 접근 O(1)
  - 코드가 간결함

## 시간/공간 복잡도

- 시간: O(n)
- 공간: O(n)

## JS 구현 포인트

- `map.get(key) || 0` 패턴으로 초기값 처리
- 객체를 해시처럼 활용
- Object.entries로 순회  
  → [key, value] 형태로 처리
- 마지막 탐색에서 `count > 0` 조건으로 명확성 확보

#### Object vs. Map

| 기준      | Object                  | Map                     |
| --------- | ----------------------- | ----------------------- |
| 기본 용도 | 단순 key-value          | 명확한 해시맵           |
| 키 타입   | 문자열 / Symbol         | **모든 타입 가능**      |
| 성능      | 빠름                    | 비슷하거나 더 안정적    |
| 순회      | 불편 (`Object.entries`) | **직관적 (`for...of`)** |
| 키 충돌   | 가능성 있음             | 없음                    |
| 메서드    | 없음                    | `.get()`, `.set()`      |
